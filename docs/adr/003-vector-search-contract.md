# ADR 003: Контракт векторного поиска

Дата: 2026-03-17  
Статус: Принято

## Контекст

Проекту нужен однозначный контракт векторного поиска для первой версии.

К моменту принятия решения уже были согласованы:
- хранение эмбеддингов в `knowledge_chunks.embedding`;
- использование PostgreSQL с `pgvector`;
- возврат `ChunkSearchResult` из `IVectorStore.Search`;

Нужно было явно определить:
- по какому полю выполняется поиск;
- какую метрику близости использовать;
- какой оператор `pgvector` и какой индекс применять;
- что означает `Score` в результате поиска;
- как сортировать результаты;
- нужен ли порог отсечения в первой версии.

## Решение

Мы фиксируем следующий контракт векторного поиска:

- Поиск выполняется по `knowledge_chunks.embedding`.
- Метод `Search` принимает эмбеддинг запроса и `limit`.
- Для первой версии значение `limit` по умолчанию равно `5`.
- Поиск использует cosine distance (косинусовое расстояние).
- Для cosine distance в SQL используется оператор `pgvector` `<=>`.
- Для индекса HNSW используется `vector_cosine_ops`.
- SQL-запрос сортирует результаты по расстоянию в порядке возрастания.
- Чтобы использовать индекс, запрос должен содержать `ORDER BY` по оператору расстояния и `LIMIT`.
- `ChunkSearchResult.Score` содержит cosine similarity, то есть `1 - cosine distance`.
- Чем больше `Score`, тем результат релевантнее.
- Метод возвращает `limit` наиболее близких чанков вместе с документом и `Score`.
- На первом этапе порог отсечения не используется.

## Альтернативы

- Использовать L2 distance.
  Это допустимо, но хуже соответствует ожидаемой семантике
  для смысловой близости текстовых эмбеддингов в первой версии.
- Использовать inner product.
  Это может быть быстрее для нормализованных векторов,
  но требует отдельной договорённости о нормализации эмбеддингов.
- Возвращать в `Score` расстояние, а не similarity.
  Это упростило бы SQL, но ухудшило бы читаемость прикладного кода:
  там удобнее правило «больше — лучше».
- Ввести порог отсечения сразу.
  Это полезно позже, но для первой версии добавляет лишнюю настройку
  без достаточного количества реальных вопросов.

Текущий вариант выбран как самый простой и понятный для `v1`.

## Последствия

- Положительные:
  поиск, индекс и интерпретация `Score` становятся однозначными;
  прикладной код получает привычную шкалу релевантности:
  больше — лучше;
  значение `limit` по умолчанию согласовано с настройками RAG;
  реализация поиска остаётся простой для первой версии.
- Негативные:
  без порога отсечения поиск всегда возвращает до `limit` результатов,
  даже если они слабо релевантны;
  сортировка для использования индекса идёт по distance,
  а прикладной `Score` считается отдельно;
  выбор cosine distance зафиксирован до появления реальных данных
  для сравнения с другими метриками.
- Что нужно сделать дальше:
  отразить это правило в `IVectorStore` и архитектурной документации;
  реализовать индекс `vector_cosine_ops`;
  реализовать SQL-запросы с `ORDER BY ... <=> ... LIMIT ...`;
  покрыть поиск тестами.
