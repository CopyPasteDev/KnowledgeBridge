# ADR 001: Стартовый технический стек v1

Дата: 2026-03-13  
Статус: Принято

## Контекст

Проекту нужен согласованный стартовый технический стек для первой версии внутреннего RAG-ассистента. На старте важно сохранить низкий порог входа для локальной разработки, не завязать слой приложения на конкретные SDK и при этом оставить путь к более стабильной конфигурации для рабочего контура без переписывания доменной логики.

Система должна:
- индексировать данные из YouTrack;
- искать релевантные фрагменты в векторной базе;
- отвечать на вопросы через RAG API и Telegram-бота;
- оставаться понятной, расширяемой и пригодной для постепенной замены локальных компонентов на внешние сервисы.

Дополнительные ограничения:
- на старте предпочтителен локальный сценарий работы;
- доступ к YouTrack должен быть сервисным, а не пользовательским;
- ingestion должен поддерживать идемпотентную переиндексацию;
- выбор провайдеров должен задаваться конфигурацией, а не кодом.

## Решение

Мы выбираем следующий стартовый технический стек для v1:

- Мы будем использовать Ollama как стартовую среду запуска для LLM и генерации эмбеддингов.
- Мы будем использовать `qwen2.5:7b` как основную модель для генерации ответов и `llama3.2:3b` как более лёгкий локальный резервный вариант.
- Мы будем использовать `nomic-embed-text` как модель эмбеддингов по умолчанию и разрешим переключение на `nomic-embed-text-v2-moe` через конфигурацию.
- Мы будем держать интеграции с LLM и генерацией эмбеддингов за явными абстракциями, чтобы слой приложения не зависел от конкретного провайдера.
- Мы будем использовать PostgreSQL с расширением `pgvector` как хранилище для векторного поиска.
- Мы будем хранить данные минимум в двух таблицах: `source_documents` и `knowledge_chunks`.
- Мы будем использовать постоянный токен от отдельной сервисной учётной записи для доступа к YouTrack.
- Мы будем считать резервным вариантом для рабочего контура `gpt-5-mini` для основной модели и `text-embedding-3-small` для эмбеддингов, если локальная схема не подтвердит качество или эксплуатационную устойчивость.

## Альтернативы

- Сразу использовать внешние LLM и сервисы эмбеддингов.
  Подходит, если важнее качество и предсказуемость среды, чем стоимость и автономность. Минусы: выше операционная и финансовая зависимость от внешних API уже на старте.
- Жёстко привязаться к одному локальному стеку без провайдерных абстракций.
  Подходит для короткого прототипа. Минусы: быстро создаёт технический долг и усложняет переход на другие модели и провайдеров.
- Хранить данные для векторного поиска в одной таблице без разделения на документы и чанки.
  Реализация проще на старте, но хуже поддерживает идемпотентный ingestion, переиндексацию и трассировку чанков к источнику.

Текущий вариант выбран потому, что он даёт рабочую стартовую конфигурацию для локальной разработки, не закрывает путь к резервным вариантам для рабочего контура и поддерживает изоляцию инфраструктурных решений от доменной логики.

## Последствия

- Положительные:
  локальная разработка остаётся дешёвой и быстрой;
  провайдерные абстракции снижают стоимость последующей замены Ollama на внешние сервисы;
  разделение `source_documents` и `knowledge_chunks` упрощает переиндексацию, трассировку источников и поддержку векторного поиска;
  сервисная учётная запись в YouTrack снижает зависимость от персональных учётных записей.
- Негативные:
  локальные модели могут не подтвердить нужное качество или приемлемую задержку ответа;
  появляется дополнительная сложность в конфигурации провайдеров и хранении секретов;
  стек хранения в PostgreSQL требует отдельной инфраструктурной реализации и настройки.
- Что нужно сделать дальше:
  подготовить начальную инфраструктуру БД: docker-compose, миграции или SQL-скрипты.
