# ADR 004: Доступ к PostgreSQL из кода через EF Core и Pgvector.EntityFrameworkCore

Дата: 2026-03-17  
Статус: Принято

## Контекст

Проекту нужно зафиксировать способ доступа к PostgreSQL из кода,
чтобы дальше последовательно реализовывать:
- схему БД;
- `IVectorStore`;
- миграции;
- интеграционные тесты для записи и поиска.

К моменту принятия решения уже были согласованы:
- использование PostgreSQL с расширением `pgvector`;
- разделение данных на `source_documents` и `knowledge_chunks`;
- семантика `upsert`;
- контракт векторного поиска на cosine distance.

Нужно было явно определить:
- через какой стек приложение работает с PostgreSQL;
- чем управлять схемой БД и транзакциями;
- как подключить тип `vector` и операции `pgvector` в коде;
- где проходит граница между слоем приложения и конкретным ORM.

## Решение

Мы фиксируем следующий способ доступа к PostgreSQL из кода:

- Основной путь доступа к PostgreSQL в `KnowledgeBridge.Infrastructure` строится на `EF Core`.
- Провайдером PostgreSQL является `Npgsql.EntityFrameworkCore.PostgreSQL`.
- Для типа `vector`, индексов `pgvector` и операций расстояния используется `Pgvector.EntityFrameworkCore`.
- Слой приложения продолжает работать только через абстракцию `IVectorStore` и не зависит от `DbContext`.
- `DbContext`, конфигурация моделей, миграции и SQL-специфика PostgreSQL остаются внутри инфраструктурного слоя.
- Доменные модели не становятся контрактом хранения один в один; инфраструктура при необходимости использует собственные persistence-модели.
- Основной путь эволюции схемы БД для проекта первой версии — миграции EF Core.
- Если отдельный запрос неудобно или невыгодно выражать через LINQ, точечный SQL допускается, но только внутри инфраструктуры и без обхода `IVectorStore`.

## Альтернативы

- Использовать чистый `Npgsql` и писать SQL вручную для всех операций.
  Это даёт максимальный контроль над SQL, но увеличивает объём шаблонного кода,
  усложняет сопровождение схемы и делает миграции менее удобными для первой версии.
- Использовать `Dapper` вместе с `pgvector`.
  Это уменьшает накладные расходы ORM и остаётся ближе к SQL,
  но не даёт такого же цельного пути для модели БД, миграций и конфигурации схемы.
- Использовать EF Core без `Pgvector.EntityFrameworkCore`.
  Это оставило бы нас с ручной интеграцией типа `vector` и специальных операторов,
  хотя для них уже есть специализированная библиотека.

Текущий вариант выбран как наиболее прагматичный для `v1`:
он даёт единый путь для модели, миграций, транзакций и интеграции `pgvector`,
при этом оставляет возможность использовать точечный SQL там, где это действительно нужно.

## Последствия

- Положительные:
  схема БД, транзакции и ограничения получают единый технический фундамент;
  интеграция `pgvector` остаётся в привычной для .NET форме;
  прикладной слой не зависит от деталей ORM;
  реализацию `IVectorStore` можно строить поверх одного `DbContext`.
- Негативные:
  появляется дополнительный слой конфигурации EF Core;
  часть поведения `pgvector` всё равно нужно будет знать на уровне PostgreSQL,
  особенно для индексов и производительности;
  для некоторых запросов может понадобиться точечный SQL внутри инфраструктуры.
- Что нужно сделать дальше:
  добавить инфраструктурный `DbContext` и регистрацию провайдера;
  описать таблицы и ограничения в конфигурации моделей;
  подготовить миграции или стартовый bootstrap БД;
  реализовать `IVectorStore` и покрыть его интеграционными тестами.
