# ADR 002: Семантика upsert для source_documents и knowledge_chunks

Дата: 2026-03-17  
Статус: Принято

## Контекст

Проекту нужен однозначный контракт записи данных в векторную базу.
Без него нельзя стабильно реализовать `IVectorStore`, схему БД и тесты.

К моменту принятия решения уже были согласованы:
- две таблицы: `source_documents` и `knowledge_chunks`;
- идемпотентный ingestion;
- полная переиндексация чанков при изменении контента.

Нужно было явно определить:
- что является внутренним и внешним идентификатором документа;
- по какому ключу ищется документ при `upsert`;
- что входит в `content_checksum`;
- что обновляется при неизменившемся и изменившемся контенте;
- как трактуется `updated_at`;
- какие ограничения нужны для чанков.

## Решение

Мы фиксируем следующую семантику `upsert`:

- `source_documents.id` — внутренний идентификатор документа и первичный ключ.
- `source_documents.id` генерирует приложение как `UUID v7`.
- `source_documents.external_id` — внешний идентификатор документа и `UNIQUE` ключ.
- Документ при `upsert` ищется по `external_id`.
- `content_checksum` считается только по `title + нормализованный текст`.
- Если документа нет, создаётся новая запись документа и вставляется весь набор чанков.
- Если документ найден и `content_checksum` совпадает, обновляется только `updated_at`.
- Если документ найден и `content_checksum` изменился, обновляются поля документа, старые чанки удаляются, новый набор чанков вставляется заново.
- `metadata` обновляется только вместе с изменением контента.
- `updated_at` хранит время записи в БД.
- Время изменения данных в источнике хранится в `metadata`.
- `knowledge_chunks.id` генерируется заново при каждой переиндексации.
- Вся операция `upsert` выполняется в одной транзакции.
- Метод записи не возвращает статус вставки или обновления.
- Для чанков вводится ограничение `UNIQUE (source_document_id, chunk_index)`.

## Альтернативы

- Обновлять документ и чанки при каждом `upsert`, даже если checksum не изменился.
  Это проще по семантике, но создаёт лишние записи и нагрузку на БД.
- Обновлять `metadata` даже при совпавшем checksum.
  Это делает данные источника свежее, но ломает простое правило:
  метаданные и контент обновляются одним и тем же событием.
- Обновлять чанки частично, а не полной заменой.
  Это потенциально экономит записи, но заметно усложняет код,
  транзакционность и тестирование первой версии.
- Не вводить `UNIQUE (source_document_id, chunk_index)`.
  Это упростило бы схему, но не защищало бы порядок чанков на уровне БД.

Текущий вариант выбран как самый простой и предсказуемый для первой версии.

## Последствия

- Положительные:
  семантика записи стала однозначной для кода, схемы БД и тестов;
  `external_id` отвечает за идемпотентный поиск документа;
  checksum позволяет пропускать лишнюю переиндексацию;
  полная замена чанков упрощает реализацию и отладку;
  ограничение по `(source_document_id, chunk_index)` защищает порядок чанков.
- Негативные:
  при совпавшем checksum метаданные источника не обновляются;
  полная замена чанков делает запись дороже, чем частичное обновление;
  приложению нужно явно генерировать `UUID v7`.
- Что нужно сделать дальше:
  отразить это правило в `IVectorStore`;
  отразить его в архитектурной документации;
  закрепить схему ограничениями и транзакцией в миграциях или SQL;
  покрыть поведение тестами.
