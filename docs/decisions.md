# Технические решения

## 1. Назначение

Этот документ фиксирует стартовые технические решения для v1 и запасные продакшен-варианты на случай, если local-first конфигурации окажется недостаточно.

---

## 2. LLM Runtime

Стартовый выбор:

- Runtime: Ollama

Причины:

- local-first подход для разработки и ранней валидации
- простая операционная модель
- отсутствие зависимости от внешних платных API на старте

Ограничение:

- все интеграции с LLM и embeddings должны быть реализованы через явные абстракции провайдеров, чтобы Ollama не был жёстко зашит в application layer

---

## 3. LLM-модель

Стартовые варианты:

- `qwen2.5:7b`
- `llama3.2:3b`

Решение:

- использовать `qwen2.5:7b` как основную модель по умолчанию ради более высокого качества ответов
- оставить `llama3.2:3b` как более лёгкий запасной вариант для ограниченных окружений или более быстрой локальной итерации

Правило выбора:

- имя модели должно задаваться через конфигурацию приложения
- предположения, завязанные на конкретную модель, не должны протекать в доменную логику или orchestration prompt-ов

Продакшен-фолбэк:

- `gpt-5-mini`

Когда переключаться:

- если качество локального Ollama окажется недостаточным на реальных вопросах команды
- если latency, требования к железу или операционная стабильность сделают локальную схему непрактичной

---

## 4. Модель эмбеддингов

Стартовые варианты:

- `nomic-embed-text`
- `nomic-embed-text-v2-moe`

Решение:

- начать с `nomic-embed-text` как базового варианта
- разрешить переключение на `nomic-embed-text-v2-moe` через конфигурацию, если он покажет лучшее качество retrieval на этапе оценки

Правило выбора:

- провайдер embeddings и модель должны конфигурироваться независимо от chat-модели
- генерация эмбеддингов для ingestion и для пользовательского запроса должна использовать одну и ту же настроенную embedding-модель в рамках одного индекса

Продакшен-фолбэк:

- `text-embedding-3-small`

Когда переключаться:

- если качество retrieval окажется недостаточным
- если локальная генерация эмбеддингов будет слишком медленной для переиндексации или использования во время запроса

---

## 5. Аутентификация в YouTrack

Решение:

- использовать permanent token, выпущенный для отдельного service account

Причины:

- исключает привязку системы к личной учётной записи сотрудника
- упрощает управление правами и аудит
- снижает операционные риски при изменении состава команды

Ограничения:

- токен должен храниться в конфигурации через secret management или переменные окружения
- service account должен иметь только минимально необходимые права на чтение индексируемых задач и комментариев
- система не должна предполагать интерактивный OAuth или auth flow, привязанный к пользовательской сессии

---

## 6. Storage

Решение:

- использовать PostgreSQL с `pgvector`
- использовать минимум две таблицы вместо одной `knowledge_chunks`:
  - `source_documents`
  - `knowledge_chunks`

Причины:

- отделяет метаданные исходного документа от векторных данных чанков
- упрощает идемпотентный ingestion и частичную переиндексацию
- упрощает отслеживание актуальности документа, идентичности источника и происхождения чанков

Ожидаемое разделение ответственности:

- `source_documents` хранит одну запись на каждую индексируемую исходную сущность
- `knowledge_chunks` хранит текстовые чанки и эмбеддинги, связанные с исходным документом

Минимальное направление схемы:

- `source_documents`: source system, внешний id, title, snapshot нормализованного контента или checksum, metadata, timestamps обновления
- `knowledge_chunks`: ссылка на исходный документ, текст чанка, порядок чанка, embedding, metadata, timestamps обновления

Ограничение:

- каждая запись чанка должна однозначно трассироваться до родительского исходного документа

---

## 7. Принципы конфигурации

- все выборы провайдеров должны настраиваться без изменения кода
- application services должны зависеть от интерфейсов, а не от SDK конкретных провайдеров
- переключение с Ollama на hosted production providers не должно требовать изменений в доменной логике

---

## 8. Текущий стек по умолчанию

Для первоначальной реализации использовать:

- Runtime: Ollama
- LLM: `qwen2.5:7b`
- Локальная fallback LLM: `llama3.2:3b`
- Embeddings: `nomic-embed-text`
- Опциональная локальная альтернатива для embeddings: `nomic-embed-text-v2-moe`
- YouTrack auth: permanent token от отдельного service account
- Storage: PostgreSQL + `pgvector`

Продакшен-фолбэк, если local-first схема окажется недостаточной:

- LLM: `gpt-5-mini`
- Embeddings: `text-embedding-3-small`
